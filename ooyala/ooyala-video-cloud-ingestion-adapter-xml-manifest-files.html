Ooyala-Video Cloud: Archivos de manifiesto XML

Puede continuar usando FTP, Aspera o MRSS con un manifiesto XML para transferir sus videos a Video Cloud. Este tema explica la estructura del XML.
Nota: este tema se aplica solamente a los clientes de Ooyala migrados a Video Cloud.

Introducción

Al ingerir videos, puede usar archivos de manifiesto XML para especificar metadatos adicionales (incluido el título del video, la descripción, los tiempos de vuelo, las etiquetas y los metadatos personalizados), así como los archivos de miniaturas y subtítulos opcionales. Puede utilizar archivos de manifiesto XML cuando ingiera contenido con FTP, Aspera o MRSS.

Nota: Este tema se aplica a OIS v2.5.

Manejo de metadatos

Si incluye metadata para los videos cuando los ingiere, se asigna a los campos de video de Video Cloud de una de estas dos formas:

  1. De forma predeterminada, cada par metadata clave/valor se añade como etiqueta en la forma: meta::key::value. Por ejemplo, <ooyala:metadata name="category">cooking</ooyala:metadata> produciría la etiqueta meta::category::cooking
  2. Si prefieres usar custom_fields para almacenar metadatos, Brightcove Support puede cambiar una configuración en su cuenta para hacerlo. Tenga en cuenta lo siguiente:
    • Deberá crear campos personalizados con nombres que correspondan a sus claves de metadatos.
    • El número de campos personalizados está limitado a 10 de forma predeterminada, por lo que si necesita más para acomodar sus metadatos, deberá solicitar campos personalizados adicionales al Soporte.
    • Si un campo personalizado correspondiente a cada elemento de metadatos no existe en su cuenta, la operación de metadatos completa fallará.

Elementos del manifiesto XML

Puede utilizar los siguientes elementos en el archivo de manifiesto XML.

Elementos del manifiesto XML
Etiqueta XML Descripción Requerido / Opcional
<channel> Contiene uno o más <item> elementos. requerido
<item> Representa un solo video para ingerir. Puede especificar varios videos (<item> etiquetas) en un archivo de manifiesto XML. Contiene las etiquetas restantes en esta tabla. requerido
<Contenido multimedia> Especifica un video para ingerir. Uno de los siguientes:
  • Nombre de archivo de un video que subiste.
  • Para la ingestión a través de MRSS (consulte Ingesta de un feed de MRSS alojado de forma remota), la URL donde se encuentra el video. La URL puede incluir http: // o https: //.
Para un activo remoto, use <ooyala: activo remoto> en lugar de.		 requerido (excepto para activos remotos)
<medios: título> Nombre o título del video. opcional
<media: miniatura> Especifica una miniatura (imagen de vista previa). Uno de los siguientes:
  • Nombre de archivo de una miniatura que cargó.
  • URL donde se encuentra la miniatura. La URL puede incluir http: // o https: //.
 opcional
<media: subTitle> Especifica un archivo de subtítulos cerrados (compatible con archivos DFXP, WebVTT y SRT). Ver Ingesta de archivos de subtítulos ocultos. Atributos:
  • href (obligatorio): uno de los siguientes:
    • Nombre de archivo del archivo de subtítulos que cargó. Ejemplo: subtítulo-1.dfxp
    • URL donde se encuentra el archivo de subtítulos. La URL puede incluir http: // o https: //.
    lang (opcional): idioma del archivo de subtítulos. Código de dos letras. Ver Subtítulos opcionales admitidos.
  • cuadros por segundo (opcional): velocidad de fotogramas del archivo de subtítulos cerrados. Las unidades están en cuadros por segundo (FPS). El valor predeterminado es 30. Los archivos de subtítulos cerrados más antiguos pueden tener 24.
Nota: Si se especifica, es su responsabilidad proporcionar valores válidos para lang y cuadros por segundo. Los valores válidos se agregan al archivo DFXP. Los valores no compatibles se ignoran. De cualquier manera, el archivo de subtítulos se agrega a Backlot y se vincula al activo.
 opcional
<medios: descripción> Descripción del video. opcional
<Enlace> URL permanente donde incrusta el video. Se asigna a los activos creados alojado_en propiedad. opcional
<términos dc:válido> Horas de vuelo que representan las horas de inicio y finalización en las que se puede reproducir el activo. Ejemplo: 
  <Términos DC:Valid>start=2011-06-28T 15:15:00-07:00; final = 2022-12-01T 16:00:00-07:00; esquema = W3C-DTF </dcterms:válido >
 opcional
<ooyala: etiquetas> Una o más etiquetas, separadas por comas. Ejemplo: / deportes, / deportes / trotar. Para conocer los antecedentes, consulte Etiquetas. opcional
<medios: palabras clave> Otra forma de definir etiquetas (sin barra diagonal). Los contenidos, separados por comas, se agregarán como etiquetas al video. Se creará una nueva etiqueta solo si aún no existe una etiqueta con el mismo nombre.

Por ejemplo, para lo siguiente <medios: palabras clave> etiqueta, las etiquetas etiqueta 1 y etiqueta 2 se crearía y agregaría al video como etiquetas. Una etiqueta llamada etiqueta de los padres también se crearía, con la etiqueta etiqueta infantil anidado en su interior.

  <medio: palabras clave> etiqueta 1, etiqueta 2, etiqueta padre/etiqueta infantil</medio: palabras clave >
Nota: Los contenidos de <medios: palabras clave> son ignorados cuando el <ooyala: etiquetas> se incluye en el archivo de manifiesto XML o en el MRSS.
 opcional
<ooyala: código de inserción> Identificación de video de Video Cloud. opcional (obligatorio si está actualizando un video en lugar de agregar uno nuevo)
<ooyala: profileguid> ID del perfil de procesamiento, que es un grupo de codificaciones que definen los formatos a los que se debe convertir un video maestro durante la transcodificación. Comuníquese con su representante de soporte de Ooyala para obtener más detalles. opcional
<ooyala: metadatos> Metadatos personalizados para el video. Puede tener un elemento para cada tipo de metadatos. Ver el Manejo de metadatos sección para obtener detalles sobre cómo metadata se agrega a los videos como etiquetas especiales. Ejemplo: <ooyala:metadata name="category">cooking</ooyala:metadata> opcional
<ooyala: activo remoto> Especifica un activo remoto para ingerir. Un activo remoto es un video que está alojado en otro lugar que no sea Backlot. Especifique los siguientes atributos:
  • durationInMs: duración del medio (en milisegundos)
  • url: URL del activo remoto. La URL puede incluir http: // o https: //.
Esta configuración se asigna al valor de la alojado_en propiedad para un activo remoto. Ejemplo: <ooyala:duración remoteasset inMS='5000' url=» http://your_website/your_remote_assets.mp4 "/>
Nota: Una vez que se ha creado un activo remoto, su propagación a las distintas CDN puede demorarse 60 segundos o más. Si solicita un activo remoto demasiado pronto después de su creación, los CDN almacenarán en caché los resultados, lo que puede tardar varios minutos en borrarse. Práctica recomendada: después de la creación, espere 30 o 60 segundos, consulte con el API de backlot [GET] /v2/activos/asset_id route, y después de recuperar el código de inserción del activo remoto (ID de contenido o ID de activo), proceda a insertar el activo.
 requerido (solo activos remotos)
<ooyala: profileguid> Especifica un perfil de ingesta de Video Cloud que se utilizará para procesar el vídeo. Si no desea usar el perfil predeterminado de la cuenta, debe especificar el que desea usar en su lugar. Ver Gestión de perfiles de ingesta. opcional

Paso a paso

Creación de un archivo de manifiesto XML

Para crear un archivo de metadatos XML:

  1. Sube tus videos y otro contenido (miniaturas, archivos de subtítulos, etc.).
  2. Abra un editor de texto, preferiblemente uno que valide XML.
  3. Agregue los encabezados XML necesarios. Por ejemplo: 
      <? xml version="1.0" encoding="utf-8"? > <rss version="2.0" xmlns:media=» http://search.yahoo.com/mrss/ "xmlns:dcterms=» http://purl.org/dc/terms/ "xmlns:fh=» http://purl.org/syndication/history/1.0 "xmlns:ooyala=» http://www.ooyala.com/mrss/ «>
  4. Crear un canal envase.
  5. Crear un artículo entrada para cada video, que contiene los elementos deseados de la tabla anterior.

    En el siguiente ejemplo, el XML especifica metadatos para dos videos. Para el primer video, el archivo y las miniaturas se encuentran en un servidor de carga y se extraen de ese servidor. Para el segundo video, el archivo y la miniatura ya se cargaron en un servidor Ooyala. El segundo video también tiene horarios de vuelo.

      <? xml version="1.0" encoding="utf-8"? > <rss version="2.0" xmlns:media=» http://search.yahoo.com/mrss/ "xmlns:dcterms=» http://purl.org/dc/terms/ "xmlns:fh=» http://purl.org/syndication/history/1.0 "xmlns:ooyala=» http://www.ooyala.com/mrss/ «> <channel> <item> <media:contenido url=» http://mysite.com/upload/lacrosse_70.mov "/> <media:miniatura url=» http://mysite.com/upload/lacrosse_70_previewimage.jpg "/> <Media:title>Deportes que realmente existen: Lacrosse</media:title > <Media:Description>Mi descripción de mi vídeo</media:description > <ooyala:etiquetas>/sports/lacrosse,/ hobbies/lacrosse</ooyala:labels > <link> http://mysite.com/videos/sports/real_sports.html</link > <ooyala:metadata name=» video_id">70</ooyala:metadatos > <ooyala:metadata name="season_number">1</ooyala:metadata > </item > <item> <media:content url=» curling_71.mov "/> <media:miniatura url=» curling_71_previewimage.jpg "/> <Media:título>Deportes que realmente existen: Curling</media:title > <media:Description> Mi descripción de mi vídeo</media:description > <ooyala:etiquetas>/sports/curling,/ hobbies/curling</ooyala:etiquetas > <link> http://mysite.com/videos/sports/real_sports.html</link > <ooyala:nombre de metadatos =» video_id">71</ooyala:metadatos > <ooyala:metadata name="season_number">1</ooyala:metadata > <! — establecer tiempos de vuelo —> <DCTERMS:Valid>start=2011-06-28T 15:15:00-07:00; end=2022-12-01T 16:00:00-07:00; scheme=W3C-DTF </dcterms:valid > </item > </channel > </rss >
  6. Guarde el archivo como XML. Asegúrese de especificar un nombre exclusivo. Si especifica dos archivos con el mismo nombre y carga uno mientras se procesa el otro, es posible que se ignore el segundo.
  7. Con el método de carga que elija, cargue el archivo de manifiesto XML a medida que cargaba los videos. Ver cualquiera FTP o Aspera.
